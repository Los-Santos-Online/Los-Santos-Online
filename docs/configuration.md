# Configuration

[Back to documentation index](README.md)

The public `.env.example` intentionally contains only 12 settings. Copy it to
`.env`, replace the placeholders, and never commit the resulting file.

## Core settings

| Variable | Description |
| --- | --- |
| `DOMAIN` | Public hostname without a scheme. It drives ROS URLs and first-time TLS certificate generation. |
| `SERVER_SECRET` | One long URL-safe secret used for private PostgreSQL authentication and signed upload tokens. |

Generate `SERVER_SECRET` with:

```sh
docker run --rm node:22-alpine node -e "console.log(require('node:crypto').randomBytes(32).toString('hex'))"
```

Docker fixes the database user/name to `lso`, builds internal PostgreSQL URLs, and
keeps PostgreSQL private. You do not configure database URLs, TLS paths, internal
ports, certificate names, or upload-token settings.

## Saves and stats R2 bucket

| Variable | Description |
| --- | --- |
| `S3_ENDPOINT` | R2/S3 endpoint. |
| `S3_ACCESS_KEY` | Saves/stats bucket access key. |
| `S3_SECRET_KEY` | Saves/stats bucket secret key. |
| `S3_BUCKET_NAME` | Saves/stats bucket name. |

## UGC R2 bucket

| Variable | Description |
| --- | --- |
| `S3_UGC_ENDPOINT` | R2/S3 endpoint for UGC. It may match `S3_ENDPOINT`. |
| `S3_UGC_ACCESS_KEY` | UGC bucket access key. |
| `S3_UGC_SECRET_KEY` | UGC bucket secret key. |
| `S3_UGC_BUCKET_NAME` | UGC bucket name. |

An R2 endpoint normally looks like:

```text
https://<ACCOUNT_ID>.r2.cloudflarestorage.com
```

Fixed object prefixes are `ps3`, `ps4`, `ps5`, `xbox360`, `xboxone`, and `pc`.
Saves, stats, and profile images are written to R2 rather than the container
filesystem.

## Discord

| Variable | Description |
| --- | --- |
| `DISCORD_TOKEN` | Optional player-count bot token. Leave blank to disable Discord. |
| `DISCORD_GUILD_ID` | Optional guild for immediate `/playercount` registration; blank registers globally. |

The bot only displays active player counts in its presence and answers the
`/playercount` command. It has no channel IDs, channel posting, message monitoring,
or message-content intent.

## Values derived automatically

- `ROS_HOST` comes from `DOMAIN`.
- The TLS certificate CN comes from `DOMAIN`. Its SAN list contains both `DOMAIN`
  and `*.DOMAIN` (for example, `dev.lossantosonline.com` and
  `*.dev.lossantosonline.com`).
- TLS remains enabled on internal port 443 with files under `certs/tls`.
- HTTP remains enabled on internal port 80.
- PostgreSQL user, database, hostname, and URLs are generated by Compose.
- Photo upload URLs use `DOMAIN`.
- Photo upload tokens use `SERVER_SECRET` and expire after 900 seconds.
- BlueSphere uses its built-in verification endpoint and timeout defaults.
- OpenSSL is pinned to 3.4.1 by the Dockerfile.
- The external relay advertised to clients is configured by `RELAY_PUBLIC_ADDRESS`
  and `RELAY_PUBLIC_PORT`. Docker does not run or expose a relay server.

## Advanced deployment overrides

These are deliberately absent from `.env.example`. Add one only when the default
deployment truly needs it:

| Variable | Default | Purpose |
| --- | --- | --- |
| `HTTP_PUBLIC_PORT` | `80` | Host HTTP port. |
| `HTTPS_PUBLIC_PORT` | `443` | Host HTTPS port. |
| `BLUESPHERE_VERIFY_URL` | Built-in production URL | Alternate BlueSphere validation endpoint. |
| `BLUESPHERE_VERIFY_TIMEOUT_MS` | `5000` | Validation timeout. |

The checked-in example intentionally preserves `51.81.86.255:6677`. These values
identify an external relay; they do not expose a Docker port.
